# 模型接入与视觉辅助方案

## 当前结论

项目不采用“只让大模型直接看图判断”的路线，而采用：

```text
地图图斑上下文 parcel_context
    ↓
代码辅助视觉预检
    ↓
多模态模型图文理解
    ↓
规则库约束推理
    ↓
结构化输出与评测
```

这样做的原因是：图斑审查需要可解释证据、稳定字段、规则追溯、空间上下文和低成本批量评测。多模态模型擅长理解图片现状，但不能仅凭照片确认法定地类、权属人、种植人或真实空间位置。

正式业务逻辑应当是：

```text
地图/业务系统给定固定图斑、权威地类、图层来源、责任主体
    ↓
外业人员上传该图斑现场图片
    ↓
模型核查图片现状是否与图斑上下文冲突
```

## 模型职责边界

模型应该做：

- 看图理解现场内容，例如建筑、硬化、道路、植被、堆料、施工痕迹、遮挡等。
- 结合文字说明和 `parcel_context` 判断是否疑似冲突。
- 根据 R001-R006 规则输出 `rule_hits`。
- 生成可解释 `evidence` 和 `reason`。
- 在证据不足、空间信息不足、图像缺失或规则冲突时建议人工复核。

模型不应该做：

- 仅凭照片确认该地块法定地类。
- 仅凭照片确认该图斑属于谁、谁在种植或使用。
- 擅自推翻地图图层、图斑属性或业务系统导入的上下文。
- 在没有坐标、图斑边界或地图要素 ID 时假装知道真实空间位置。

推荐表达方式：

```text
在地图/业务系统给定该图斑为耕地的前提下，图片中疑似出现建筑物或硬化地表，因此命中 R002。
```

不推荐表达方式：

```text
模型确认该照片所在地一定是耕地。
```

## 推荐主模型路线

当前推荐继续沿 Qwen-VL 系列推进。考虑当前预算较低、第一阶段重点是跑通链路，推荐顺序为：

1. 原型验证：`Qwen/Qwen2.5-VL-3B-Instruct` 本地开源模型。
2. API 验证：DashScope/百炼 OpenAI-compatible 图文接口。
3. 效果增强：Qwen3-VL 或更强视觉语言模型。
4. 辅助证据增强：OpenCV、YOLO、SAM、GIS 空间校验。

保留 `MockModelClient` 的原因：

- 不消耗 API 预算。
- 不依赖 GPU。
- 可以稳定跑测试和批量评测。
- 可以验证规则、Prompt、Parser、Web/API 是否被改坏。

## 已完成工程落地

### 1. 视觉预检层

新增模块：

```text
src/modules/vision_precheck/
```

当前能力：

- 判断图片路径是否为空。
- 判断图片文件是否存在。
- 判断图片文件是否可读取。
- 若本机安装 Pillow，可读取图片尺寸、长宽比和 EXIF 信息。
- 输出稳定 JSON 字段，写入 `metadata.vision_precheck`。

### 2. Qwen-VL API 客户端

新增文件：

```text
src/modules/model_gateway/qwen_vl_client.py
```

当前支持：

- 从配置文件读取模型名、API base、key 环境变量、超时、重试等参数。
- 使用图文消息格式构造请求。
- 本地图片存在时自动转为 base64 data URL。
- 未设置 API key 时，在联网前明确报错。

启用方式：

```bat
set DASHSCOPE_API_KEY=你的DashScopeKey
set DACHUANG_MODEL_CONFIG=configs\model.example.json
python run_demo.py
```

### 3. 本地 Qwen2.5-VL 开源模型客户端

新增文件：

```text
src/modules/model_gateway/local_qwen_vl_client.py
configs/model.local_qwen25vl.example.json
```

当前支持：

- 默认模型：`Qwen/Qwen2.5-VL-3B-Instruct`。
- 配置化切换 provider：`qwen2_5_vl_local`、`local_qwen_vl` 等。
- 使用 Transformers + `qwen-vl-utils` 进行本地图文推理。
- 图片存在时作为 image 输入；图片缺失时仍可用文本和规则跑通流程。
- 未安装依赖时给出清晰错误，不影响 Mock 冒烟测试。

注意：当前机器检测到 PyTorch 为 CPU 版本，未检测到 CUDA。CPU 跑 3B 视觉语言模型会比较慢，因此本地模型更适合作为开源路线保留；真实效果验证优先用 API 小样本。

### 4. 配置化 pipeline

入口：

```python
build_pipeline_from_config(config_path)
```

默认不传配置时使用 `MockModelClient`。设置 `DACHUANG_MODEL_CONFIG` 后可切换真实模型或本地开源模型。

### 5. 通用模型审查 API

当前 Web 服务暴露：

```text
GET  /api/v1/health
POST /api/v1/inspect
POST /api/inspect
```

推荐 JSON 请求结构：

```json
{
  "parcel_id": "FJ-2026-001",
  "image_base64": "...",
  "image_filename": "parcel.jpg",
  "parcel_context": {
    "authoritative_land_type": "耕地",
    "land_type_source": "三调属性",
    "map_layer": "三调地类图层",
    "map_feature_id": "feature-001",
    "spatial_reference": "图斑边界",
    "parcel_location": "福建省某县某村",
    "land_user": "某种植人或合作社",
    "right_holder": "某责任主体"
  },
  "text_description": "现场举证照片用于核查当前土地利用近况。"
}
```

也支持 `multipart/form-data` 上传图片，适合普通网页表单。

## 后续扩展路线

### 阶段 A：真实图片小样本验证

先准备 3-6 张真实或脱敏图片，覆盖：

- 正常耕地
- 耕地疑似建筑占用
- 林地疑似硬化或堆料
- 建设用地用途异常
- 图像缺失或证据不足

每条样本都应补齐 `parcel_context`。

### 阶段 B：轻量代码证据

优先完善：

- EXIF 拍摄时间与任务时间对比。
- EXIF GPS 与图斑位置对比。
- 图片是否过小、模糊、过暗、不可读。
- 上传图片是否缺少元数据。

### 阶段 C：传统视觉辅助

后续可接：

- OpenCV：模糊、过暗、边缘、纹理、颜色占比等质量与场景线索。
- Rasterio/GDAL/Shapely：遥感图、GeoTIFF、图斑边界、空间裁剪和坐标一致性。
- YOLO：建筑、道路、施工机械、堆料区等目标检测。
- SAM 2：道路、硬化区域、植被区域等分割辅助。

### 阶段 D：真实模型评测

用同一套样本对比：

- Mock 模型结果
- Qwen-VL API 结果
- 本地 Qwen2.5-VL 结果（如果机器性能允许）
- 每条规则 recall / precision
- 失败样本原因分析

## 当前注意事项

- API key 只能放环境变量，不能写入仓库、文档或聊天记录。
- 模型不能替代地图图层、图斑边界、权属台账或业务系统。
- `parcel_context` 是权威上下文，图片只是核查现场近况。
- 当前真实图片样本不足，真实模型效果还需要小样本验证。
- 发布前需要重新生成 `private_bundle.zip.dcb`，否则受保护源码改动不会进入公开仓库。
