# 模型接入与视觉辅助方案

## 当前结论

项目不采用“只让大模型直接看图判断”的路线，而采用：

```text
代码辅助视觉预检
    ↓
多模态模型图文理解
    ↓
规则库约束推理
    ↓
结构化输出与评测
```

这样做的原因是：图斑审查需要可解释证据、稳定字段、规则追溯和低成本批量评测。传统代码适合提取确定性证据，多模态模型适合综合图像、文本、元数据和规则做最终判断。

## 推荐主模型

当前推荐第一优先接入 Qwen-VL 系列。考虑当前预算较低、第一阶段重点是跑通链路，推荐顺序调整为：

1. 原型验证：`Qwen/Qwen2.5-VL-3B-Instruct` 本地开源模型。
2. 效果增强：`Qwen3-VL-4B-Instruct`、`Qwen3-VL-8B-Instruct` 或百炼 API。
3. 批量稳定评测：保留 Mock，使用同一套样本对比 Mock、本地模型和 API 模型。

推荐原因：

- 第一阶段不是追求榜单最强，而是验证“图片输入 -> 模型理解 -> 规则 JSON -> 后端保存 -> 前端展示”。
- `Qwen/Qwen2.5-VL-3B-Instruct` 参数较小，适合低成本本地原型验证。
- Qwen2.5-VL 具备图像理解、目标定位、OCR、结构化输出等能力，贴合 `evidence` 和 `rule_hits`。
- 后续可以平滑切换到 Qwen3-VL 或百炼 API，形成从开源原型到增强模型的路线。

API 方式仍然保留，适合后续效果增强：

- `Qwen3-VL`：适合后续视觉理解、空间关系、OCR、目标定位等能力扩展。
- `Qwen2.5-VL`：适合作为成熟可用的视觉语言模型路线。
- API 侧可先用兼容 OpenAI Chat Completions 结构的服务，降低工程接入成本。

推荐原因：

- 中文业务规则理解更贴合项目场景。
- 更适合在大创材料中表达“国产多模态模型接入”。
- 后续可从 API 迁移到 ModelScope 或本地开源权重。
- 当前项目已有 `ModelClient.inspect(prompt, image_path)` 抽象层，替换模型成本低。

## 已完成的工程落地

### 1. 视觉预检层

新增模块：

```text
src/modules/vision_precheck/
```

当前能力：

- 判断图片路径是否为空。
- 判断图片文件是否存在。
- 判断图片文件是否可读取。
- 若本机安装 Pillow，可读取图片尺寸、长宽比和 EXIF 信息。
- 输出稳定 JSON 字段，写入 `metadata.vision_precheck`。

当前输出示例：

```json
{
  "image_exists": false,
  "image_readable": false,
  "signals": ["image_file_missing"],
  "quality_flags": []
}
```

注意：当前不会因为样本图片暂缺而阻断流程，只会把图片状态作为辅助证据传给后续 Prompt。

### 2. Qwen-VL 客户端骨架

新增文件：

```text
src/modules/model_gateway/qwen_vl_client.py
src/modules/model_gateway/factory.py
```

当前支持：

- 从配置文件读取模型名、API base、key 环境变量、超时、重试等参数。
- 使用图文消息格式构造请求。
- 本地图片存在时自动转为 base64 data URL。
- 未设置 API key 时，在联网前明确报错。

### 3. 本地 Qwen2.5-VL 开源模型客户端

新增文件：

```text
src/modules/model_gateway/local_qwen_vl_client.py
configs/model.local_qwen25vl.example.json
```

当前支持：

- 默认模型：`Qwen/Qwen2.5-VL-3B-Instruct`。
- 配置化切换 provider：`qwen2_5_vl_local`、`local_qwen_vl` 等。
- 使用 Transformers + `qwen-vl-utils` 进行本地图文推理。
- 图片存在时作为 image 输入；图片缺失时仍可用文本和规则跑通流程。
- 未安装依赖时给出清晰错误，不影响 Mock 冒烟测试。

启用方式：

```bat
set DACHUANG_MODEL_CONFIG=configs\model.local_qwen25vl.example.json
python run_demo.py
```

首次本地运行前需要安装依赖：

```bat
pip install torch torchvision transformers accelerate pillow qwen-vl-utils
```

### 4. 配置化 pipeline

新增入口：

```python
build_pipeline_from_config(config_path)
```

默认仍使用 `MockModelClient`，保证本地测试和演示稳定。

如果要启用真实模型，可设置：

```bat
set DACHUANG_MODEL_CONFIG=configs\model.example.json
set DASHSCOPE_API_KEY=你的DashScopeKey
```

然后运行：

```bat
python run_demo.py
```

或：

```bat
python run_web.py
```

## 后续扩展路线

### 阶段 A：轻量代码证据

优先完善：

- 图片尺寸、分辨率、长宽比异常。
- EXIF 拍摄时间与任务时间对比。
- EXIF GPS 与图斑位置对比。
- 图片是否过小、不可读、缺少元数据。

### 阶段 B：传统视觉辅助

后续可接：

- OpenCV：模糊、过暗、边缘、纹理、颜色占比等质量与场景线索。
- Rasterio/GDAL/Shapely：遥感图、GeoTIFF、图斑边界、空间裁剪和坐标一致性。
- YOLO：建筑、道路、施工机械、堆料区等目标检测。
- SAM 2：道路、硬化区域、植被区域等分割辅助。

### 阶段 C：多模态模型综合推理

本地 Qwen2.5-VL 或 API 版 Qwen-VL 负责：

- 看图理解现场内容。
- 结合文本说明和地类信息。
- 参考 `metadata.vision_precheck` 中的代码证据。
- 根据 R001-R006 规则输出合法 JSON。

### 阶段 D：真实模型评测

用同一套 26 条样本进行对比：

- Mock 模型结果
- Qwen-VL 真实模型结果
- 每条规则的 recall / precision
- 失败样本原因分析

## 当前注意事项

- 真实 API key 只能放环境变量，不能写入仓库。
- 当前图片路径仍以占位为主，因此真实模型效果评测还需要后续补图。
- `MockModelClient` 仍然保留，用于无 API key 时跑测试和演示。
- Qwen-VL 客户端骨架已经完成，但真实 API 调用需要设置 `DASHSCOPE_API_KEY` 后再验证。
