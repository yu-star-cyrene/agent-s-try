# 材料梳理与当前架构

## 项目定位

本项目不是泛化聊天机器人，而是面向国土图斑审查场景的垂直预检系统。当前系统目标是融合地图图斑上下文、现场举证图片、文本说明、元数据和业务规则，先做一轮机器预审，提前筛出明显异常、疑似异常和需要人工复核的样本。

当前架构已经从“手动填写地类 + 上传图片”的演示模式，推进为更接近正式业务系统的“地图图斑上下文 + 上传图片核查”模式。正式使用时，地类、图斑边界、图层来源、种植人或责任主体应由 GIS/地图前端或业务系统提供；模型只负责核查上传图片中的土地近况是否与这些权威上下文冲突。

## 现实痛点

- 外业举证强依赖人工，效率低。
- 福建山地丘陵地形复杂，图斑碎片化、隐蔽化明显。
- 存在异地拍照、旧图新用、死角漏拍、避重就轻等造假风险。
- 单靠人工比对，难以同时处理图像内容、空间线索、图斑属性和业务规则。
- 如果缺少图斑边界、坐标、地类来源和人员台账，模型不能仅凭照片确认真实地类或权属关系。

## 当前系统流程

```text
GIS/地图前端或业务系统
    ↓
传入 parcel_context
    ↓
上传现场举证图片
    ↓
代码辅助视觉预检
    ↓
加载结构化规则库 R001-R006
    ↓
构建规则驱动 Prompt
    ↓
模型推理（Mock / Qwen-VL API / 本地 Qwen2.5-VL）
    ↓
解析结构化 JSON
    ↓
生成 evidence、rule_hits 和人工复核建议
    ↓
Web 展示或 API 返回给其他前端
```

## parcel_context 设计

`parcel_context` 是当前架构中最重要的上下文字段，代表地图图层、图斑属性、权属/种植人或业务系统导入的权威信息。

推荐结构：

```json
{
  "authoritative_land_type": "耕地",
  "land_type_source": "三调属性",
  "map_layer": "三调地类图层",
  "map_feature_id": "feature-001",
  "spatial_reference": "图斑边界",
  "parcel_location": "福建省某县某村",
  "land_user": "某种植人或合作社",
  "right_holder": "某责任主体"
}
```

模型不能把 `parcel_context` 当成自己推断出来的结论。它只能在该上下文前提下判断图片现状是否疑似冲突，例如：

```text
在地图/业务系统给定该图斑为耕地的前提下，上传图片中疑似出现建筑物或硬化地表，因此存在耕地疑似建筑占用风险。
```

不能写成：

```text
模型确认该照片所在地一定是耕地。
```

## 当前已完成内容

### 1. 规则库结构化

规则库已从简单文本升级为结构化规则，包含规则类别、风险等级、适用地类、触发条件、所需证据、判断逻辑、输出口径、优先级和启用状态。

对应文件：

- `data/rules/rule_template.json`
- `data/rules/rule_schema.json`

### 2. 样本集与标准答案

当前已经建立 26 条规则覆盖样本。每条样本包含图斑编号、图片路径、地类信息、文本说明、元数据、标准答案和命中规则 ID。

对应文件：

- `data/samples/sample_set.json`
- `data/samples/sample_schema.json`
- `data/samples/request_example.json`

### 3. Prompt 推理链路

`prompt_builder.py` 已经读取结构化规则字段，并加入地图图斑上下文约束。模型必须基于规则库、图片线索和 `parcel_context` 输出合法 JSON。

### 4. 代码辅助视觉预检

`src/modules/vision_precheck/` 会在调用多模态模型前提取图片存在性、可读性、尺寸、长宽比、EXIF 时间和 GPS 等低成本证据，并写入 `metadata.vision_precheck`。

### 5. 模型接入层

当前支持三类模型入口：

- `MockModelClient`：本地流程验证和测试。
- `QwenVLClient`：通过 DashScope/百炼等 OpenAI-compatible 图文接口调用真实模型。
- `LocalQwenVLClient`：通过 `Qwen/Qwen2.5-VL-3B-Instruct` 做低成本本地开源模型验证。

对应文件：

- `src/modules/model_gateway/mock_client.py`
- `src/modules/model_gateway/qwen_vl_client.py`
- `src/modules/model_gateway/local_qwen_vl_client.py`
- `src/modules/model_gateway/factory.py`

### 6. Web 与通用模型审查 API

当前 Web 页面已经不只是演示台，也提供可复用模型审查接口：

```text
GET  /api/v1/health
POST /api/v1/inspect
POST /api/inspect
```

接口支持：

- `multipart/form-data` 上传图片。
- JSON 传 `image_path`。
- JSON 传 `image_base64`。
- 传入完整 `parcel_context`。
- 返回统一结构化 JSON。

这意味着后续其他已经写好的前端页面，只需要调用 `/api/v1/inspect`，不需要复制模型、Prompt、规则库或 Parser 逻辑。

### 7. 结构化输出与规则追溯

输出结果包含：

- `is_abnormal`
- `issue_type`
- `reason`
- `confidence`
- `requires_manual_review`
- `evidence`
- `rule_hits`
- `raw_output`

其中 `rule_hits` 是后续答辩、评测和人工复核追溯的重要字段。

### 8. 批量评测闭环

当前批量评测入口：

```bat
python scripts\evaluate_samples.py
```

输出：

- `outputs/reports/evaluation_summary.json`
- `outputs/reports/evaluation_details.json`
- `outputs/reports/evaluation_details.csv`

当前 Mock 环境下 26 条样本可以稳定跑通，R001-R006 规则级指标保持 1.0。注意：这只代表流程闭环正确，不代表真实多模态模型最终效果。

## 当前限制

- 当前真实图片样本不足，很多样本仍以占位图片路径为主。
- 真实 API 已能跑通单条调用，但还需要真实/脱敏图片验证视觉判断效果。
- `parcel_context` 已有接口设计，但还没有接入真实 GIS 数据、图斑边界文件或人员台账。
- Schema 文件已经存在，但 `parcel_context` 的 Schema 还可以继续细化。
- OpenCV、YOLO、SAM、GIS 空间校验等辅助证据层仍属于后续增强。

## 后续重点

1. 用新版 Web 或 `/api/v1/inspect` 上传真实/脱敏图片，验证 `parcel_context + image` 的真实模型输出。
2. 准备 3-6 条小规模真实样本，覆盖耕地、林地、建设用地、正常样本和证据不足样本。
3. 补充真实或模拟的地图图层字段、图斑要素 ID、种植人/责任主体字段。
4. 对 Mock 与真实模型在同一批样本上做规则级对比评测。
5. 根据真实失败样本优化 Prompt、Parser 和人工复核提示。
6. 将 API 调用示例、Web 截图、评测表格整理成中期检查或答辩材料。
